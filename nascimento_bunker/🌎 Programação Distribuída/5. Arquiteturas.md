## Introdução
A ideia de "arquitetura" em desenvolvimento de software diz mais sobre "como deve ser feito" do que qualquer outra coisa, digamos que são "mapeamentos" mentais de como a aplicação deve ser dividida, não é regra escrita em pedra mas sim um guia padronizado de como grandes projetos se mantém até os dias atuais.
### 1. Arquiteturas em camadas
Responsável por dividir o sistemas em níveis com responsabilidade bem definidas.
- Apresentação
- Aplicação
- Dados
Exemplos comuns são serviços MVC para aplicações web. Organizado e de fácil manutenção.

### 2. Arquitetura Orientada à Serviços
Serviços independentes que se comunicam entre si. 
- Serviços reutilizáveis em N contextos dentro de uma mesma aplicação
- Comunicação feita via rede
Exemplo pode ser um sistema corporativo integrando vários sistemas. Reusabilidade e integrável.

### 3. Arquitetura baseada em objetos
Objetos distribuídos que interagem remotamente, chamando métodos em objetos que estão em outras máquinas.
- RMI
- CORBA
Abstração forte e reuso de código, porém alto nível de acoplamento.

### 4. Arquitetura baseada em Recursos
Baseada no conceito de recursos HTTP. 
- GET, POST, PUT, DELETE.
Amplamente utilizadas em API moderna como Twitter. Simples, escalável e amplamente adotado.

### 5. Publicação e Assinatura
Comunicação baseada em eventos.
- Produtor publica mensagem
- Consumidores são notificados ou as recebem diretamente.
Utilizado para construção de web-hooks como Kafka ou RabbitMQ.

### Quando usar cada uma?
1. Use **Camadas** quando for sistema simples ou bem estruturado
2. Use **SOA** quando integração entre sistemas grandes
3. Use **Objetos distribuídos** quando contextos legados (hoje menos comum)
4. Use **REST** quando APIs web modernas
5. Use **Pub/Sub** quando sistemas reativos e escaláveis
